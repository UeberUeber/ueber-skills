# Brainstorm-ing 예시 워크스루

## 문제: "이 React 앱의 온보딩 이탈률이 80%다"

---

# Part 1: 브레인스토밍

## Step 1: 문제정의

유저: "온보딩에서 사람들이 다 나가요"

Claude가 5요소로 정리:

| 요소 | 내용 |
|---|---|
| 현상 | 신규 사용자 80%가 온보딩 중 이탈 |
| 이해관계 | 서비스가 좋아도 사용자가 경험하지 못함. 성장 병목 |
| 목표 상태 | 온보딩 완료율 50% 이상 |
| 불변 제약 | ? |
| 열린 공간 | ? |

Claude: "불변 제약이 뭐가 있나요? 예를 들어 백엔드는 못 건드린다거나."
유저: "백엔드는 건드릴 수 있어. 근데 기존 유저 데이터 구조는 바꿀 수 없어."
Claude: "바꿔도 되는 건 뭐가 있나요?"
유저: "UI 전부 다 바꿔도 돼. 온보딩 단계도 자유롭게."

→ 문제 v1 완성:

| 요소 | 내용 |
|---|---|
| 현상 | 신규 사용자 80%가 온보딩 중 이탈 |
| 이해관계 | 성장 병목. 서비스 경험 불가 |
| 목표 상태 | 온보딩 완료율 50% 이상 |
| 불변 제약 | 기존 유저 데이터 구조 변경 불가 |
| 열린 공간 | UI 전체, 온보딩 단계/흐름, 백엔드 |

---

## Step 2: 페르소나 선정

Claude가 3축 삼각형으로 직접 선정:

| | A: Don Norman | B: Dieter Rams | C: Shigeru Miyamoto |
|---|---|---|---|
| 인지 스타일 | 데이터 기반 관찰 | 원리에서 연역 | 직감 + 플레이테스트 |
| 지식 영역 | UX 심리학, 어포던스 | 미니멀 디자인 10원칙 | 게임 레벨 디자인 |
| 관점 | 사용자 편의 최적화 | 불필요한 것 제거 | 재미 + 학습 최적화 |
| 삼각형 | 문제에 딱 맞는 전문가 | A의 정반대 (추가가 아닌 제거) | A-B와 다른 분야 (게임) |

---

## Step 3: 독립 발산

3개 Task 병렬. 각 페르소나가 4단계 발산 + Verbalized Sampling.

**Norman 에이전트 (발췌)**:

1단계 — 자유 생성:
```
1. [28%] 단계 수 5→3으로 줄이기
2. [22%] 프로그레스 바 추가
3. [15%] 소셜 로그인으로 입력 최소화
4. [8%] 각 단계에 "왜 이걸 묻는지" 설명
5. [5%] 자동 저장으로 이탈 후 복귀 지원
```

2단계 — 반대 방향:
```
6. [3%] 온보딩을 아예 안 보여주고 나중에 필요할 때만
7. [2%] 단계를 오히려 늘리되 각 단계를 극도로 단순하게
...
```

3단계 — 7기법 적용, 4단계 — 메타 생성 (생략)

**Rams 에이전트**: 온보딩 삭제, 빈 대시보드, 가입 없이 사용 등
**Miyamoto 에이전트**: 퀘스트 온보딩, 안개 맵, 달성 배지 등

→ 3명 합산 약 100개 아이디어

---

## Step 3.5: 클러스터링

Claude가 직접 처리:

**클러스터**:
- A) 마찰 제거 — 단계 축소, 소셜 로그인, 자동 저장
- B) 온보딩 제거 — 바로 대시보드, 필요할 때만 수집
- C) 게이미피케이션 — 배지, 퀘스트, 레벨 디자인

**모순 쌍**:
- Rams "바로 대시보드" ↔ Miyamoto "튜토리얼 = 첫 번째 실제 작업"
  → 정반대지만 공통 통찰: "온보딩과 사용의 경계를 없앤다"
- Norman "단계를 늘려서 각각 단순하게" ↔ Rams "단계를 전부 없애라"

---

## Step 4: 교차 수분 (Teams)

```
TeamCreate("bs-onboarding")
→ 3명 스폰, 전체 아이디어 + 클러스터 + 모순 쌍 전달
```

Norman (교차 조합):
> "Rams의 '바로 대시보드'와 내 '프로그레스 바'를 합치면:
> [4%] 대시보드 자체가 프로그레스 바 역할. 기능을 쓸수록 대시보드가 채워진다"
→ SendMessage to Miyamoto

Miyamoto (연쇄 반응):
> "오, 그러면 빈 대시보드가 게임의 안개 맵(fog of war)!
> [2%] 기능 발견 = 맵 개방. 탐험 동기 부여"
→ SendMessage to Rams

Rams (비판적 빌드업):
> "안개 맵이면 온보딩 '단계'가 아예 필요 없음.
> [1%] 사용자가 자연스럽게 탐험하면서 온보딩 완료"
→ SendMessage to all

→ **연쇄 반응으로 "안개 맵 온보딩" 컨셉 탄생** (어느 한 명의 아이디어가 아님)

```
shutdown → TeamDelete
```

---

## Step 5: 개인 통합

3개 Task 병렬. 각 페르소나가 교차 수분 결과를 받고 혼자 소화.

Norman:
> 통합: "안개 맵 + 내 UX 전문성 → 구체적 인터랙션 설계. 기능별 힌트 타이밍, 첫 사용 시 피드백 루프"
> 빈 곳: "접근성 — 시각 장애 유저에게 안개 맵은 어떻게?"
> 새 아이디어: "[1%] 안개 맵 대신 '성장하는 정원' 메타포 — 기능 사용 = 식물 성장"

Rams:
> 통합: "안개 맵의 핵심 = 점진적 노출. 미니멀 원칙과 일치. 처음엔 핵심 3개만 보이고 나머지는 안개"
> 새 아이디어: "[2%] 안개가 아니라 '화이트 스페이스' — 사용할수록 공간이 채워지는 디자인"

Miyamoto:
> 통합: "안개 맵 + 퀘스트를 합치면: 탐험 퀘스트 = 기능 발견. 보스전 = 핵심 워크플로우 완주"
> 새 아이디어: "[1%] 협동 온보딩 — 기존 유저가 신규 유저와 함께 탐험"

## Step 6: 수렴

### 6-1. 클러스터링 (Claude)

| 방향 | 핵심 | 대표 아이디어 | 강점 | 리스크 |
|---|---|---|---|---|
| 마찰 제거 | 기존 온보딩 개선 | 단계 축소 + 소셜 로그인 | 안전, 점진적 | 근본 해결 아님 |
| 안개 맵 | 대시보드 = 탐험 = 온보딩 | 점진적 노출 + 기능 발견 | 혁신적, 교차에서 탄생 | 구현 복잡도 |
| 게이미피케이션 | 보상 루프로 동기 부여 | 배지 + 퀘스트 + 협동 | 재미, 리텐션 | 과하면 역효과 |

### 6-2. 독립 평가 (3 Task 병렬)

Norman 평가 (발췌):
> 안개 맵: 문제 적합성 5/5, 독창성 5/5, 실현 가능성 3/5, 확장성 4/5

Rams 평가:
> 안개 맵: 문제 적합성 4/5, 독창성 5/5, 실현 가능성 4/5, 확장성 5/5

Miyamoto 평가:
> 안개 맵: 문제 적합성 5/5, 독창성 4/5, 실현 가능성 3/5, 확장성 3/5

### 6-3. 종합 + 유저 선택

Claude:
> 안개 맵이 3명 모두에게 높은 평가. 실현 가능성에서 의견 차이 (Norman/Miyamoto: 3, Rams: 4).
> 마찰 제거는 안전하지만 근본 해결이 아니라는 점에서 3명 모두 낮게 평가.

유저: "안개 맵으로 간다. 마찰 제거의 소셜 로그인도 합치자. **실행.**"
